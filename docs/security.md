# Security Guidelines

Rules for contributors implementing new features in this codebase.

---

## SQL Injection

### Order clauses
User-supplied `order` strings are sanitized at two layers before reaching SQL:

1. **Syntax** â€” `CallKwArgs::order()` (`core/interfaces/IViewModel.hpp`) hand-parses the string and rejects anything that isn't `col [ASC|DESC]` tokens separated by commas. Throws `std::invalid_argument` on violation.
2. **Column existence** â€” `BaseModel::validateOrder_()` (`modules/base/BaseModel.hpp`) checks every column name against the model's `FieldRegistry`. Throws on unknown columns, preventing information leakage through sort order on unexposed fields.

**Rule:** Never pass a raw `call.order()` string directly to any SQL query. Always let it flow through `BaseModel::search()` or `searchRead()` which call `validateOrder_()` automatically.

### Domain (WHERE) clauses
`Domain::toSql()` (`modules/base/Domain.hpp`) uses pqxx parameterised queries for all values and validates column names against `sanitizeColumn_()`. Values are never string-concatenated.

**Rule:** Never build WHERE clauses by string concatenation. Use `domainFromJson()` + `toSql()`.

### Field names in write() / SELECT
`write()` and `buildSelectCols_()` both guard field name concatenation with `fieldRegistry_.has()` before touching the name in SQL. The registry is populated only from C++ `registerFields()` code.

**Rule:** When adding new write paths, always check `fieldRegistry_.has(key)` before using the key in SQL. Never trust JSON object keys as column names without this check.

## Session Management

Session IDs are 128-bit tokens generated by `RAND_bytes()` (`core/infrastructure/SessionManager.hpp`). Never replace this with `std::rand`, `mt19937`, or any other PRNG.

Cookies are set `HttpOnly`, `SameSite=Lax`, `Path=/`. Do not relax these attributes.

`SameSite=Lax` is load-bearing for CSRF protection on same-site deployments â€” it prevents the session cookie from being sent on cross-origin POST requests.

## Authentication and Rate Limiting

The `authenticate` endpoint is rate-limited by `LoginRateLimiter` (`core/infrastructure/JsonRpcDispatcher.hpp`): 10 failed attempts per IP per 5-minute window.

**Rule:** If you add any other credential-accepting endpoint (password reset, API key validation), apply the same pattern â€” check `rateLimiter_.allow(ip)` before processing, call `recordFailure` / `recordSuccess` after.

The current limiter is in-memory. For multi-process deployments, move the state to PostgreSQL or Redis.

## CORS

`HttpConfig::corsOrigin` defaults to `""` â€” no `Access-Control-Allow-Origin` header is sent. This is correct for same-origin deployments (frontend served from the same server).

Set `cors_origin` in `config/system.cfg` **only** when the frontend is on a different origin (e.g. a dev server). Use the exact origin string (`http://localhost:3000`). **Never use `*` in production** â€” it exposes all API responses to third-party sites.

## Authorization (Known Gap)

There is currently no model-level or record-level access control. Any authenticated user can call any ViewModel method on any record. This is acceptable for single-user or trusted-network deployments.

**Before exposing this system to multiple untrusted users**, implement an RBAC layer:
- `ir_model_access` table: per-model read/write/create/unlink per group
- `ir_rule` table: per-record domain filters
- Group membership resolved from session uid â†’ `res_groups_users_rel`

Check this at the top of `BaseModel::search()`, `searchRead()`, `create()`, `write()`, and `unlink()`.

## Error Responses

Exception detail is gated by `HttpConfig::devMode` (default `false`). In production, clients receive a generic message; `e.what()` is logged server-side via `LOG_ERROR` only.

**Rule:** Never add a new `catch` block that returns `e.what()` directly. Always use the `devMode` ternary pattern already present in `HttpServer.hpp`. In `config/system.cfg`, set `dev_mode = true` only on local developer machines.

## Security Headers

All JSON responses carry: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, `Content-Security-Policy: default-src 'none'`. These are applied by `HttpServer::applySecurityHeaders_()` inside every handler helper.

**Rule:** Any new handler registered via `addJsonPost` / `addJsonPostWithResponse` / `addJsonGet` inherits these headers automatically. Handlers registered directly via `drogon::app().registerHandler()` must call `applySecurityHeaders_()` manually.

**HSTS** should be added at the reverse proxy (nginx/caddy), not the application â€” the application cannot determine whether it is behind TLS termination.

**Static files** served by Drogon's `setDocumentRoot()` bypass `applySecurityHeaders_()`. For production, configure the reverse proxy to add the appropriate headers to static responses, or add a Drogon response filter.

## Session Cookie

Session cookies are set with `HttpOnly=true`, `SameSite=Lax`, `Path=/`. For HTTPS deployments, the `Secure` flag is set when `HttpConfig::secureCookies = true`.

**Rule:** In `config/system.cfg`, always set `secure_cookies = true` in any deployment served over HTTPS. Never disable `HttpOnly` or `SameSite`.

## Input Validation Checklist

When adding a new endpoint or ViewModel method:

- [ ] Domain/WHERE: use `domainFromJson()` + parameterised `toSql()`
- [ ] ORDER BY: route through `BaseModel::search()` / `searchRead()`
- [ ] Field names in write: guard with `fieldRegistry_.has()` (safe because `FieldRegistry::add()` validates names at boot)
- [ ] New model field names: only ASCII identifiers â€” `FieldRegistry::add()` enforces `[a-zA-Z_][a-zA-Z0-9_]*` at startup
- [ ] String fields written to DB: bound as pqxx params, never concatenated
- [ ] New credential endpoint: apply `LoginRateLimiter` pattern
- [ ] New `catch` block: use `devMode` ternary, never return `e.what()` directly
- [ ] New handler via `drogon::app().registerHandler()` directly: call `applySecurityHeaders_()` manually
- [ ] New model with sensitive data: document the authorization requirement
